Leçon 1 : Preuve et Complexité Algorithmique

Version avancée pour portfolio académique : Cours ALC2101

Section 01

Journal d'apprentissage réflexif

Cette première leçon m'a permis de comprendre que l'algorithmique n'est pas seulement une discipline de programmation mais une science formelle. J'ai pris conscience que tester un algorithme ne constitue pas une preuve de correction.

La distinction entre terminaison et correction partielle m'a obligé à structurer ma pensée de manière mathématique. La notion d'invariant de boucle représente pour moi un tournant : elle formalise rigoureusement ce que l'on suppose intuitivement vrai dans une boucle.

La classification P/NP m'a également ouvert une perspective théorique sur les limites fondamentales du calcul. Mon principal défi reste l'automatisation de la construction d'invariants pertinents.

Section 02

Synthèse conceptuelle approfondie

2.1 Fondement de la preuve d'algorithme

La correction totale d'un algorithme A calculant une fonction f repose sur :

Théorème : Correction totale
Correction totale = Terminaison + Correction partielle
▸ Terminaison

Un algorithme termine si aucune exécution n'est infinie.

Définition : Variant

Pour les algorithmes itératifs, on identifie un variant V tel que :

  • V ∈ ℕ
  • V ≥ 0
  • V décroît strictement à chaque itération

Cela garantit qu'après un nombre fini d'itérations, V atteint 0.

▸ Correction partielle

On utilise un invariant de boucle I. Un invariant doit satisfaire :

Définition : Invariant de boucle
  • Initialisation : I vrai avant la première itération
  • Conservation : I vrai avant itération i ⇒ I vrai après
  • Conclusion : I + condition de sortie ⇒ propriété finale

Cela correspond à une preuve par récurrence :

I(0) et I(k) ⇒ I(k + 1)
2.2 Analyse mathématique d'un invariant

Prenons l'algorithme de somme :

s ← 0
Pour i ← 1 à n
    s ← s + T[i]

Invariant proposé :

I(i) : s = Σj=1i−1 T[j]
Preuve de l'invariant
Initialisation : Avant première itération : i = 1 → somme vide = 0 → invariant vrai.
Conservation : Supposons invariant vrai pour i. Après exécution :
s = Σj=1i T[j]
Invariant conservé.
Conclusion : À la fin, i = n + 1. Donc :
s = Σj=1n T[j]
■ CQFD
Section 03

Analyse approfondie de complexité

3.1 Définition formelle du Big-O
Définition : Notation Big-O

On dit que :

T(n) = O(f(n))

si :

∃ c > 0, ∃ n₀ ∈ ℕ, ∀ n ≥ n₀ : T(n) ≤ c · f(n)

Ce n'est pas une égalité, c'est une majoration asymptotique.

3.2 Étude détaillée de l'algorithme PrefMoyen
Pour i ← 1 à n
    Pour j ← 1 à i
        a ← a + tab[j]

Nombre total d'itérations :

Σi=1n i = n(n + 1) / 2 = n² / 2 + n / 2

Donc :

T(n) = O(n²)
3.3 Comparaison asymptotique des classes

Pour n très grand :

O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ)
Classe Nom n = 10 n = 100 n = 1000
O(1) Constante 1 1 1
O(log n) Logarithmique ≈ 3 ≈ 7 ≈ 10
O(n) Linéaire 10 100 1 000
O(n log n) Quasi-linéaire ≈ 33 ≈ 664 ≈ 9 966
O(n²) Quadratique 100 10 000 1 000 000
O(2ⁿ) Exponentielle 1 024 ≈ 10³⁰ ∞

Un algorithme exponentiel devient inutilisable très rapidement.

Section 04

Perspective théorique : Classes de problèmes

Classe P

Problèmes résolus en temps polynomial : O(nk)

Exemple : tri par insertion.

Classe NP

Problèmes dont une solution candidate peut être vérifiée en temps polynomial.

NP-Complet

Problèmes à la fois :

  • NP
  • NP-difficiles

Si un seul NP-complet est résolu en temps polynomial :

P = NP

Ce qui reste aujourd'hui un problème ouvert fondamental en informatique théorique.

Section 05

Application avancée (niveau professionnel)

Dans les smart contracts (blockchain) :

  • Chaque opération = coût en gas
  • Complexité quadratique = explosion des coûts

Un double parcours de tableau en Solidity :

O(n²) → Contrat potentiellement inutilisable
Stratégies d'optimisation
  • Mapping → accès en O(1)
  • Éviter les boucles imbriquées sur les données on-chain
  • Pré-calculs off-chain quand c'est possible

La complexité algorithmique devient ici une contrainte économique réelle : chaque opération superflue coûte de l'argent.
